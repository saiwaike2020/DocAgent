# AI Agent 开发与算法工程师（效果优化方向）作业补充说明

### 1. 如果这套系统后续由你负责
    请说明你对当前系统的判断：哪些地方已经可用，哪些地方还不能放心交付，下一步你会优先处理什么，为什么。

    - 答：
        * 已经可用：
        系统的基础 RAG 已经可以用起来。借助千问 VL 的多模态能力，极其棘手的复杂财报表格已能高质量转化为 Markdown；自研的“表头继承”切分算法解决了长表格断层问题；M3E + BM25 的 RRF 双路召回保障了数值类问题的精准度；Pydantic + 正则防御清洗确保了 JSON 结构化输出的极高稳定性。 

        * 存疑：
            + 成本与并发瓶颈： 目前的 PDF 解析是串行请求多模态大模型，处理百页以上文档极慢且 API 成本不可控。
            + 检索颗粒度： 仅有 Chunk 级召回，缺少 Rerank（重排）阶段，Top-K 召回在面对相似度极高的干扰项时可能失准。另外相似度或者得分过滤也要进一步测试才能确定，详见/doc/RAG_Agent.pptx
            +

        * 下一步优先处理：
            + PDF文档解析准确性问题，增加验证机制， 如在输入PDF文档时，保存用户指定页的一些关键内容进行人工识别，用来验证VL识别后的结果；
            + 考虑如何降低token费用，提高处理速度，尤其面对几百、几千页的处理问题。
            + 增加监控机制，掌握各个环节，尤其时Agent 问答环节的与LLM交互的细节，从而更多掌握系统的运行方式等等。

### 2. 如果要给客户试用

    请说明你会先和产品、客户或技术负责人确认哪些事情；你会如何理解目标、边界、验收标准和第一版交付范围。

    - 答：
        * 确认目标与边界：
        客户真实要处理的文档是什么？（是排版规整的财报、盖章的扫描件，还是手写单据？）这是决定解析方案的前提。

        * 确认核心指标： 
        客户更看重“召回率”（宁可多给不能漏掉）还是“准确率”（只给精确答案，要不就无答案）？响应时间要求是秒级还是分钟级异步？

        * 划定第一版交付范围
        明确 V1 版本支持单文档问答，暂不支持跨文档对比分析、图表趋势推理；明确只支持特定语言或特定领域的文档。
        验收标准设定为：基于双方确认的 50 个真实问题评测集，回答准确率达到 X%，且 cited_pages 溯源准确率达到 Y%。

### 3. 如果要保证质量和稳定性

    请说明你会如何验证这套系统是否可靠；如果它在演示或生产前出现问题，你会如何排查、止损、修复和确认修复有效。
    
    - 答：
        * 系统性验证与排查
        验证可靠性： 建立基于 Ground Truth 的 Golden Dataset。分离测试检索指标（Recall@5、NDCG）和生成指标（利用另一个 LLM 充当 Judge，评估忠实度、完整度）。

        * 生产问题排查与止损
            + 止损： 核心链路加上多级 Fallback。如果大模型解析 JSON 彻底崩溃，系统直接返回标准的“抱歉，系统暂时无法提取有效信息”，而不是抛出 500 异常代码给用户。
            + 排查定位： 完善链路日志（Trace）。问题一出，立即通过 utils.logger 追溯：是解析阶段丢了表格？是 BM25 没召回？还是 LLM 拿到了上下文却发生了幻觉？
            + 修复验证： 将引发 Bad Case 的文档和提问沉淀到自动化测试集（Regression Test）中，修复后必须跑通历史测试集，确保不引入新的退化。

### 4. 如果时间有限

    请说明你会如何安排优先级、同步风险、协调资源，并保证关键目标尽量按时完成。

    - 答：
        + 优先级安排： 坚持“核心主干可用，边缘分支降级”原则。优先保底文档解析、单路向量检索和 LLM 生成；复杂的 RRF 混合检索、精细化的重排模型（Reranker）可作为 P1 甚至 P2 任务后续迭代。
        + 风险同步： 提前识别风险点（如：依赖的外部 API 随时可能限流）。建立提前预警机制，如日报Token使用量，发现异常立即报警等；
        + 集中资源：先处理优先级高的问题保证交付，如先确保PDF解析，再处理其他；

    请说明你会让 AI / Agent 帮你做哪些事情，哪些事情必须自己把关，以及你如何检查它们的输出是否可信。

### 6. 补充一个真实例子

    请结合这次作业，或结合你过去的真实经历，讲一个你主动负责、推进问题闭环、在压力下保证结果的例子。重点说明你当时怎么判断、怎么沟通、怎么推进、最后结果如何。

    - 答：
        在Orcale工作的时候，来自印度的技术支持报了一个紧急的bug。一个客户，在使用我们的Activity Analysis Framework（AAF）在HR系统中为一类新入职员工新增一个计划任务时（是一个Low code No code 的组件），设置完成保存没有问题，当触发业务流程的时候，总是报错无法执行。由于这个框架历史久远，开发的人员早已离职，我也是临时受命来处理这个bug。我首先跟技术支持沟通业务场景，共同在我们的环境中重现了bug。之后我开始一边学习用户手册，一边看设计文档，去理解用户的场景。开始的判断是我们的代码有问题，就改了代码，结果回归测试有若干用例无法通过。之后，我恢复源代码，认真的debug每种场景，去理解原来的设计实现。大概不间断的看了两天代码，通过长时间的读代码、跑case，基本对这一部分功能的实现有了一定的理解把握。判断应该是用户的设置有问题，代码没问题。下一步，就是想办法证明我的想法，拿出证据给技术支持和客户看。最后经过十几个小时的不断理解和调整，终于将用户的用例配置跑通。也就是走了从代码反推出用户如何设置业务逻辑的过程。这个过程很艰辛，但是让我整理清楚了这部份代码的实现。

### 7. 如果文档章节层级解析不对
    假设这套系统解析后发现不少文档的章节层级不准确，例如标题、条款、段落、表格归属或跨页内容关系识别错，导致后续检索和问答效果受影响。

    请说明你会如何定位问题、判断影响范围、设计修复方案，并验证修复是否有效。也可以说明短期交付时如何止损，长期如何把这类问题做得更稳定。

    - 答：
        首先找到对应的PDF文档，再将解析过程中的PDF图片、解析生成的MD文档（已实现本地缓存），对比三种文档，确定是图片问题，还是VL出错的问题。
        图片问题：提高像素；更换SDK等
        VL问题： 改进Prompt，更换VL模型等

        + 短期止损： 在 Prompt 中增加对章节结构的强约束，或者退一步，在 Chunking 阶段弱化对 Markdown 标题深度的依赖，扩大 max_chunk_size 并辅以重叠率（Overlap），确保上下文不易丢失。
        + 长期治理：
            引入基于规则的后处理脚本（Post-processor），利用正则结合特定领域的标题序号模板（如 第一章 -> #, 1.1 -> ##）进行强制清洗对齐。
            改用 LayoutLM 等专业的版面分析模型，先提取精准的逻辑树（Document Tree），再基于逻辑树进行分块。

        + 验证：  抽取或生成若干篇结构类似或者更混乱的测试文档，编写自动化断言，检查解析后的 AST（抽象语法树）或 Markdown 大纲树是否与人工标注的树结构高度一致。

### 8. 如果系统效果不稳定
    请结合你提交的系统，选一个你认为最可能影响效果的问题，说明你会如何定义评价标准、收集 bad case、定位原因、尝试优化，并判断优化是否真的有效。

    - 答：
        焦点问题：特定专业名词或数值检索遗漏（False Negative）在目前的系统中，最容易影响效果的是：用户问“A业务的营收是多少？”，但系统回答“信息不足”。

        + 定义评价标准： Recall@K（正确包含数值的 Chunk 是否进入了上下文列表）和 Answer Exact Match（提取的数值是否准确无误）。
        + 收集与定位： 提取这条 Bad Case。检查日志发现，带有该数值的 Chunk 并没有被召回。深入分析发现原因：该数值在财报一个超长表格的最底端，而 Chunking 时被一切为二，失去了表头“A业务营收”的语义标签。
        + 尝试优化： 这正是我在当前系统中实现“带表头状态机切分算法”的来源。我修改了逻辑，使得任何从表格中间截断的 Chunk，都会在开头强制拼上 *(接上表)* 和表头行。另外就是可以将表的一行作为一个chunk的主要内容，同事将章节标题、表头等都打包进去。
        + 验证优化： 将优化前后的代码跑同一套测试集进行消融实验（A/B Test）。如果长表格类型的 QA 准确率从 40% 提升到了 90%，且原有普通文本的准确率没有下降，就证明该优化不仅有效且具备泛化能力，可以合入主干代码。
    

# 工作方式补充确认

这部分不考技术细节，主要想了解你真实的工作方式。请尽量结合具体经历或你会采取的实际做法回答，不需要写长篇。


## 1. 遇到不清楚但又比较急的需求

如果产品或客户给了一个比较模糊的目标，但希望尽快看到结果，你通常会怎么处理？

请说明你会先确认什么、如何推进第一版、如何避免方向做偏。

    - 答：
        + 首先尽可能确认目标，与产品和客户沟通，搞清楚他们的目的和边界；
        + 用最简单的、快捷的mock来演示他们目标，将各方认知基本对齐；
        + 动手之前，确定几个合理的检查点，将中间结果或者子目标尽早展示给产品或者客户；

## 2. 关键节点前发现问题

如果周五晚上发现系统还有一个影响演示或交付的问题，但周一就要给客户看，你会怎么安排？

请说明你会如何判断优先级、同步风险、协调资源，以及你自己会投入到什么程度。

    - 答：
    核心原则：演示的本质是“建立信任”，止损的优先级高于“根治 Bug”。
        + 现场定性与双线策略：周五晚完成 Triage（分级）。如果是局部瑕疵（如某类特殊表格解析错位），立刻在代码里加 Try-Catch 友好提示兜底；如果是主干阻塞 Bug，立即启动 Plan A（根治） 与 Plan B（静态保底） 双线并行。
        + 落实 Plan B（静态保底）：在尝试修 Bug 前，我会在周五当晚，挑出目前系统能完美跑通的 3 个经典文档和问答路径，无剪辑录制一份 3 分钟的 MP4 演示视频，并将这几组 Ground Truth 结果做本地 Cache。确保哪怕周一早上服务器主板烧了，拿着视频和缓存也能把客户讲得明明白白。
        + 设定截止时间止损线：周五晚向技术负责人同步：“发现问题 X，已备好演示预案 B。我现在着手主攻 Plan A 修复，若最晚周日 14:00 前未突破，立刻封版切回 Plan B，绝不把技术不确定性拖入周日。”

## 3. 沟通中出现分歧

如果产品、客户或技术同事对同一个问题的理解不一致，甚至有人比较着急，你会怎么沟通？

请说明你如何保持推进节奏，同时避免沟通失控或反复拉扯。

    - 答：
    核心原则：把“观点之争”转化为“现实约束条件下的最优解选择题”。
    + 议题物理隔离：当大家争执“方案A好还是方案B好”且情绪焦躁时，我会主动打断并拉出白板，不再聊技术，而是列出当下的铁律：“强调我们的时间和成本压力。” 让所有人面对共同的客观敌人（时间/成本），而不是互相说服。
    + 提供带有标价的选项：把争论具象化为二选一的决策清单，足够的信息，必要时也有个人的意见和判断，交给业务负责人拍板
    + 多说无益：遇到纯技术理解分歧，多说无益。我会直接说：“争论 1 小时的成本，高于写个脚本跑 20 条数据。今晚我把两个逻辑各跑 20 个 Case，明早按正确率数据说话。”

## 4. 多件事情同时压过来

如果同一时间有开发、问题排查、测试验证、文档说明几件事都要推进，你会怎么安排？

请说明你会如何利用 AI / Agent 工具提高效率，以及哪些结果你一定会自己复查。

    - 答：
        核心原则：按个人状态进行单线程排期，极度压榨 AI 的体力。
        - 状态好，精力足：开发、问题排查；
        - 状态一般：测试验证和文档说明；
        - AI/Agent： 排除思路，提供灵感，审查方案和检查验证逻辑；生成文档说明；
        - 自己复查：代码提交时，要坚持Diff； 检查验证结果；
